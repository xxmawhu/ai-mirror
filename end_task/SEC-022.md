## Description

Both `cmd_cp()` and `cmd_mv()` have a TOCTOU race between the file operation (cp/rename) and the subsequent `recursive_chown()`. When the destination is a directory, the chown target is computed as `dst_path / src_path.filename()`. An attacker can replace this path with a symlink between the copy/rename and the chown, causing chown to operate on an arbitrary file.

## Attack Scenario (cmd_cp)

1. Attacker runs `sudo am cp /src/file /home/attacker/project/` where `/home/attacker/project/` is a directory
2. `cp -a` creates `/home/attacker/project/file`
3. Attacker (in parallel) removes `file` and replaces it with symlink to `/etc/shadow`
4. `recursive_chown` follows symlink, changes `/etc/shadow` ownership

## Attack Scenario (cmd_mv)

1. Similar to above but with `fs::rename` instead of `cp`
2. In the `else` branch (dst is directory): rename succeeds, then attacker swaps the result
3. In the cross-filesystem branch: `cp -a` + `fs::remove` + chown — same TOCTOU window

## Affected Code

```cpp
// src/cli/commands.cpp:270-271 (cmd_cp)
fs::path chown_target = fs::is_directory(dst_path) ? dst_path / src_path.filename() : dst_path;
if (!recursive_chown(chown_target, ai_user)) {

// src/cli/commands.cpp:349-351 (cmd_mv, cross-filesystem fallback)
fs::path chown_target = fs::is_directory(dst_path) ? dst_path / src_path.filename() : dst_path;
if (!recursive_chown(chown_target, ai_user)) {
```

## Fix Suggestion

Use file descriptor-based chown (`fchownat` with `AT_SYMLINK_NOFOLLOW`) or verify the inode hasn't changed between the operation and chown:

```cpp
// After cp/rename, stat the target to get inode
auto stat_before = fs::status(chown_target);
ino_t inode_before = /* extract inode */;
// ... do chown ...
// verify inode hasn't changed

// Better: use open + fchown with O_NOFOLLOW
int fd = open(chown_target.c_str(), O_RDONLY | O_NOFOLLOW);
if (fd < 0) { /* symlink swap detected */ }
struct stat st;
fstat(fd, &st);
// verify expected ownership
fchown(fd, ai_uid, ai_gid);
close(fd);
```
