## Description

`cmd_cp()` uses `cp -a` (archive mode) which preserves SUID/SGID bits and file capabilities. After copying, `recursive_chown(chown_target, ai_user)` changes ownership but does **not** clear the SUID bit — on Linux, `chown()` only clears SUID/SGID for regular files owned by root in some kernel versions, but not reliably for all cases, and the order matters (cp -a runs first, then chown).

An attacker can craft a SUID binary in their project, copy it with `am cp`, and the SUID bit may persist after chown, giving the ai-user a SUID binary owned by ai_user.

## Attack Scenario

1. Attacker compiles a shell wrapper: `int main() { setuid(0); system("/bin/bash"); }`
2. Marks it SUID: `chmod u+s ./evil` (as main user in their project dir)
3. Runs `sudo am cp ./evil /home/attacker/ai-user-data/`
4. `cp -a` preserves SUID bit
5. `recursive_chown` changes owner to ai_user but SUID may persist
6. ai-user executes the binary → SUID to ai_user context (not root, but full ai-user privileges including any sudoers rules)

## Affected Code

```cpp
// src/cli/commands.cpp:264
std::string cp_cmd = fmt::format("cp -a '{}' '{}'", src_path.string(), dst_path.string());
```

## Fix Suggestion

Option A: Remove `-a` flag, use `cp -r --no-preserve=mode` instead:
```cpp
std::string cp_cmd = fmt::format("cp -r --no-preserve=mode '{}' '{}'", src_path.string(), dst_path.string());
```

Option B: After `recursive_chown`, explicitly clear SUID/SGID:
```cpp
std::string chmod_cmd = fmt::format("chmod -R ug-s '{}'", chown_target.string());
```

Recommended: **Option A** is simpler and more robust.
