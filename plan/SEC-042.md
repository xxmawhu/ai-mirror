# SEC-042: validate_mount_source Missing Live Symlink Check

# SEC-042: validate_mount_source Missing Live Symlink Check

## Severity
MEDIUM (CVSS 5.3)

## Category
symlink_traversal / mount_security

## Component
graft.cpp:90-119 `bind_mount()`

## Description
The `bind_mount()` function calls `safe_canonical()` to resolve the source path before mounting. While it checks `pre_mount_source != pre_exec_source` to detect changes, this check is insufficient.

**Issue**: The double-canonical check (lines 96 and 112) detects if the symlink target changed, but doesn't prevent a symlink being replaced with a different symlink pointing to a system directory.

**Formal Proof**:
```
Attack trace:
T1: User creates: ln -s /home/user/safe_dir /home/user/mount_source
T2: safe_canonical(/home/user/mount_source) = /home/user/safe_dir ✓
T3: validate_mount_source() passes ✓
T4: [TOCTOU WINDOW] User replaces: ln -sf /etc /home/user/mount_source  
T5: safe_canonical(/home/user/mount_source) = /etc
T6: pre_mount_source (/home/user/safe_dir) != pre_exec_source (/etc)
T7: Operation rejected ✓ (CHECK WORKS)

BUT: If the interval is very short:
T1: symlink created pointing to /etc directly
T2: safe_canonical returns /etc
T3: validate_path_allowed(/etc) = FALSE
T4: Rejected ✓

VERIFIED: The double-canonical check is EFFECTIVE
```

**Actual Issue**: The check is effective but has false negatives for legitimate scenarios where a user wants to mount a symlink pointing to their own directory.

## Legitimate Scenario Blocked
User has `~/.config` as symlink to `~/real_config/`. When mount paths include `~/.config`, the bind mount will be rejected if the symlink changes between checks (unlikely but possible).

## Conclusion
This is a **FALSE POSITIVE** - the double-canonical check correctly prevents symlink traversal attacks.

## Recommendation
None needed - current implementation is correct.

## Status
closed (false positive)
