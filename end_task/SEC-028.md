# SEC-028: SSHManager Tempfile Symlink Race

## Severity
MEDIUM (CVSS 5.3)

## Category
race_condition / symlink_traversal

## Component
ssh_manager.cpp:112-137 `authorize_key()`

## Description
The `authorize_key()` function writes to a temporary file at `auth_keys + ".tmp"` before atomic rename. The temp file path is predictable and located in the ai_user's home directory.

**TOCTOU vulnerability**: An ai_user with write access to their own `.ssh` directory can:
1. Pre-create a symlink at `.ssh/authorized_keys.tmp` → `/etc/passwd`
2. When SSHManager writes the new authorized_keys content, it follows the symlink
3. The atomic rename fails, but sensitive file is corrupted

## Formal Proof
```
Pre-conditions:
- ai_user owns /home/ai_user/.ssh directory (permissions 700)
- ai_user can create symlinks in their .ssh directory

Attack trace:
T1: ai_user creates: ln -s /etc/passwd ~/.ssh/authorized_keys.tmp
T2: main_user runs: ai-mirror create (triggers authorize_key)
T3: SSHManager opens ~/.ssh/authorized_keys.tmp for writing
T4: open() follows symlink, writes to /etc/passwd
T5: rename() fails (cross-device or permission)
T6: /etc/passwd corrupted with SSH key data
```

## Affected Code Paths
- `authorize_key()` (line 112-137)
- `authorize_public_key_string()` (line 180-204)
- Both use same tempfile pattern

## Impact
Corruption of system files if symlink exists. While ai_user cannot directly modify /etc/passwd permissions, they can pre-create symlinks when their .ssh directory is created.

## Recommendation
1. Use `O_NOFOLLOW | O_EXCL` when creating the temp file
2. Use `mkstemp()` for unpredictable temp file names
3. Create temp file in a secure directory (e.g., /tmp with restricted permissions)
4. Verify temp file is a regular file before writing

## Test Case
```bash
# As ai_user (if shell access allowed):
ln -s /etc/passwd ~/.ssh/authorized_keys.tmp
# As admin:
sudo ai-mirror create /home/user/testproj
# Check:
cat /etc/passwd  # Contains SSH key data instead of user list
```
