# SEC-026: Null Byte Not In Dangerous Characters List

## Metadata

## Description
`validate_path_no_shell_metachars()` does not include `\x00` (null byte) in its dangerous characters list. When strings containing null bytes are passed to `exec_safe()`, `c_str()` truncates at the first null byte, potentially bypassing validation.

## Code Evidence
```cpp
// shell.cpp:204-206
bool validate_path_no_shell_metachars(const std::string& path) {
    static const std::string dangerous = ";`$(){}[]|&<>!\n\r";
    return path.find_first_of(dangerous) == std::string::npos;  // '\0' NOT in list!
}

// shell.cpp:119-124
ShellResult exec_safe(const std::string& file, const std::vector<std::string>& args) {
    ...
    for (auto& a : args) {
        argv.push_back(const_cast<char*>(a.c_str()));  // Truncates at '\0'
    }
    ...
}
```

## Formal Analysis
### Potential Attack Path:
1. User supplies path: `/home/user/safe\x00/../etc/passwd`
2. `validate_path_no_shell_metachars()` passes (no dangerous chars before `\x00`)
3. `PathResolver::resolve()` - std::filesystem may or may not handle null bytes correctly
4. `exec_safe()` receives the string, `c_str()` truncates to `/home/user/safe`
5. Operation proceeds on truncated path

## Verification Needed
Need runtime testing to confirm:
1. Does `std::filesystem::canonical()` handle embedded null bytes?
2. Does `std::filesystem::path` constructor preserve null bytes?
3. Do `fs::exists()`, `fs::is_directory()` work correctly with null bytes?

## Suggested Fix
```cpp
bool validate_path_no_shell_metachars(const std::string& path) {
    // Check for null byte first
    if (path.find('\0') != std::string::npos) return false;
    
    static const std::string dangerous = ";`$(){}[]|&<>!\n\r";
    return path.find_first_of(dangerous) == std::string::npos;
}
```
