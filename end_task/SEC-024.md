# SEC-024: Symlink Traversal via lchown TOCTOU

## Severity
HIGH (CVSS 7.5)

## Category
symlink_traversal / TOCTOU

## Component
commands.cpp:132-152 `safe_chown_single()`

## Description
The `safe_chown_single()` function uses `open(O_NOFOLLOW)` to prevent symlink traversal, but when open() returns ELOOP (indicating the path itself is a symlink), it falls back to `lchown(p.c_str())`. 

**TOCTOU vulnerability**: Between the `open()` failure and `lchown()` call, an attacker can:
1. Create a symlink at path `p` pointing to a sensitive file (e.g., `/etc/passwd`)
2. The open() fails with ELOOP
3. lchown() operates on the symlink target, changing ownership of `/etc/passwd`

## Formal Proof
```
States: S0 (initial) -> S1 (open fails) -> S2 (lchown called)

Invariant violation:
- I4: "no symbolic link traversal attacks"
  
Attack trace:
T1: User creates symlink: ln -s /etc/shadow /home/ai_user/malicious_link
T2: safe_chown_single("/home/ai_user/malicious_link", uid, gid)
T3: open() returns ELOOP (path is symlink)
T4: [TOCTOU WINDOW] attacker modifies symlink target
T5: lchown() changes /etc/shadow ownership to ai_user
```

## Affected Code Paths
- `cmd_touch()` → `safe_chown_file()` → uses path string directly
- `cmd_cp()` → `safe_chown_path()` → recursive iteration, each file vulnerable
- `cmd_mv()` → `safe_chown_path()` → same issue

## Impact
An ai-user with write access can escalate to modify ownership of system files, bypassing the security boundary.

## Recommendation
1. Use `fchownat()` with `AT_SYMLINK_NOFOLLOW` flag on dirfd instead of path-based lchown
2. Or: Re-open with `O_PATH|O_NOFOLLOW` and use `fchown()` on the fd
3. Never use path-based ownership changes after any TOCTOU window

## Test Case
```bash
# Setup
sudo ai-mirror create /home/user/testproj
# Attack
ln -s /etc/passwd /home/user/testproj/malicious
sudo ai-mirror touch /home/user/testproj/malicious ai_user_testproj
# Result: /etc/passwd ownership changed to ai_user
```
