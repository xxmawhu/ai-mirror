## Description

`is_path_allowed()` uses a blocklist approach — it rejects paths under `/usr`, `/bin`, `/sbin`, `/lib`, `/etc`, `/proc`, `/sys`, `/dev`, `/boot`, `/var` (SYSTEM_DIRS), and accepts everything else. This means paths under `/opt`, `/data`, `/tmp`, `/srv`, `/home/otheruser`, and any other non-blocklisted location are allowed.

This is inconsistent with `cmd_create()` which requires all paths to be under the caller's home directory. The blocklist approach is fragile — any newly created top-level directory automatically becomes allowed.

## Attack Scenario

1. User A (ai-mirror group) runs `sudo am touch /opt/important_data/file` or `sudo am cp /src/file /tmp/target`
2. `is_path_allowed()` returns true (not in SYSTEM_DIRS)
3. File operation succeeds on paths outside user A's home

**Impact**: Limited in practice because the user still needs ai-mirror group membership and sudo access, but it violates the security model where users should only operate within their own home directory.

## Invariant Violated

**I2**: "All paths must be under caller's home directory" — VIOLATED

## Affected Code

```cpp
// src/utils/shell.cpp:234-262
bool is_path_allowed(const fs::path& p, const std::string& main_user) {
    // ...
    static const std::vector<std::string> SYSTEM_DIRS = { ... };
    for (const auto& sys_dir : SYSTEM_DIRS) {
        if (canonical_path.string().rfind(sys_dir, 0) == 0) {
            return false;  // blocked
        }
    }
    return true;  // allows everything else!
}
```

## Fix Suggestion

Change from blocklist to allowlist — only allow paths under the caller's home directory:

```cpp
bool is_path_allowed(const fs::path& p, const std::string& main_user) {
    fs::path canonical_path = safe_canonical(p);
    if (canonical_path.empty()) return false;

    // Allowlist: must be under caller's home
    fs::path home_dir = fs::path("/home") / main_user;
    std::string path_str = canonical_path.string();
    std::string home_str = home_dir.string();

    if (path_str.rfind(home_str, 0) != 0) {
        return false;  // NOT under caller's home
    }
    return true;
}
```

This makes the check consistent with `cmd_create`'s behavior.
